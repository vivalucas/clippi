### ADR-001 [2025-05-14] UI 架构选型：双原生 UI + Rust 核心库

**状态**：已采用

**替代关系**：无

**背景与需求**：需要在 macOS 和 Windows 两个平台提供原生体验的 GUI，同时最大化复用核心逻辑代码，控制开发成本。

**采用的方案**：macOS 用 Swift + SwiftUI，Windows 用 C# + WinUI 3，共用 Rust 核心库（编译为 .dylib / .dll），两端 UI 通过 FFI 调用核心库。

**备选方案**：

1. Electron（跨平台 Web 技术栈）
   - 优点：一套代码跨平台，开发效率高
   - 缺点：资源占用高（内存、CPU），应用体积大，体验不够原生
   - 放弃原因：与"应用本身轻量"的核心目标冲突

2. Tauri（Rust + Web 前端）
   - 优点：比 Electron 更轻量，Rust 原生支持
   - 缺点：UI 层仍是 Web 技术，原生体验不如 SwiftUI / WinUI 3
   - 放弃原因：UI 体验不够原生，且 Web 前端在桌面场景下有局限

3. 纯 Rust GUI（如 iced、egui）
   - 优点：全 Rust 技术栈
   - 缺点：GUI 框架成熟度不足，难以实现系统原生风格
   - 放弃原因：GUI 框架生态不够成熟，难以达到 HandBrake / Permute 的体验水准

**决策依据**：原生 UI 资源占用更低，体验更好；AI 可承担两端 UI 的开发成本；Rust 核心库最大化复用逻辑代码。

**改动范围**：整体架构，影响 core/ / macos/ / windows/ 目录结构和构建流程。

### ADR-002 [2025-05-14] ffmpeg 管理策略：静态编译随应用打包

**状态**：已采用

**替代关系**：无

**背景与需求**：需要确定如何在应用中集成 ffmpeg——依赖用户系统已安装的版本，还是自行打包。

**采用的方案**：静态编译 ffmpeg，随应用打包。通过 scripts/ 脚本在开发环境和 CI 中自动下载固定版本，校验 SHA256 后使用。

**备选方案**：

1. 调用系统已安装的 ffmpeg
   - 优点：无需打包，应用体积更小
   - 缺点：版本不一致导致行为差异，用户需自行安装，增加使用门槛
   - 放弃原因：用户体验不一致，与"普通用户无需了解 ffmpeg"的目标冲突

2. 动态链接系统 ffmpeg
   - 优点：体积较小
   - 缺点：依赖系统库路径，移植性差
   - 放弃原因：跨平台兼容性问题

**决策依据**：保证所有用户体验一致，无需用户额外安装，版本可控。

**改动范围**：scripts/ 目录、.gitignore、CI/CD 配置、应用打包流程。

### ADR-003 [2025-05-14] 仓库结构：Monorepo

**状态**：已采用

**替代关系**：无

**背景与需求**：核心库和两端 UI 需要版本同步，且 AI 协作开发时需要完整项目上下文。

**采用的方案**：单仓库（Monorepo）管理 core/ + macos/ + windows/ + scripts/。

**备选方案**：

1. 多仓库（core、macos-ui、windows-ui 分开）
   - 优点：职责分离清晰
   - 缺点：版本同步困难，AI 获取上下文不便，跨仓库修改需多个 PR
   - 放弃原因：增加协调成本，不利于 AI 协作

**决策依据**：保证核心库与两端 UI 版本同步，便于 AI 协作开发时获取完整上下文。

**改动范围**：仓库结构、CI/CD 配置。

### ADR-004 [2025-05-14] 任务执行策略：串行执行

**状态**：已采用

**替代关系**：无

**背景与需求**：多文件批量处理时，需要决定是串行还是并行执行 ffmpeg 任务。

**采用的方案**：串行执行，同一时间只有一个 ffmpeg 任务在运行。

**备选方案**：

1. 并行执行多个任务
   - 优点：理论上可更快处理多文件
   - 缺点：多任务并行互相抢占 GPU/CPU 资源，总耗时可能反而更长，GPU 编码器可能不支持多实例
   - 放弃原因：资源抢占导致性能不可控

**决策依据**：避免多任务并行互相抢占 GPU/CPU 资源，ffmpeg 参数始终传 `-threads 0` 由 ffmpeg 自动决定最优线程数。

**改动范围**：Rust 核心库 queue.rs。

### ADR-005 [2025-05-14] 裁剪模式：快速模式 vs 精确模式

**状态**：已采用

**替代关系**：无

**背景与需求**：视频裁剪需要在速度和精度之间做取舍。

**采用的方案**：提供两种模式供用户选择：
- 快速模式（`-c copy`）：不重编码，速度快但首帧可能不精确（依赖关键帧位置）
- 精确模式（重编码）：帧精确但耗时更长

**决策依据**：能用 `-c copy` 的操作绝不触发重编码，但同时为需要精确裁剪的用户提供选项。

**改动范围**：Rust 核心库 task.rs，UI 裁剪模块。
